0 · Architecture & how it fits together

Before any code, hold the whole picture in your head. A request to create a link is a normal synchronous write. But the redirect path is where the interesting design lives: you must redirect the user instantly, yet you also want to record analytics. You do not want analytics work slowing the redirect. That tension is exactly what Kafka solves.

┌─────────────────────────────┐ POST /api/shorten │ Django + DRF (web) │ write ───────────────────▶│ create ShortURL row │──────────▶ ┌────────────┐ │ │ │ PostgreSQL │ GET /<code> │ look up code, 302 redirect │◀────────── └────────────┘ ───────────────────▶│ + fire-and-forget event │ read ▲ └──────────────┬──────────────┘ │ update │ produce "url-clicks" │ click_count ▼ │ ┌───────────────┐ consume ┌─────┴────────┐ │ Apache Kafka │────────────────▶ │ Consumer │ │ topic: clicks│ (own group) │ (worker pod)│ └───────────────┘ └──────────────┘ All of the above run as separate Deployments inside one Kubernetes cluster.
Why this shape The redirect handler publishes a tiny event and returns immediately — single-digit milliseconds. A completely separate consumer process reads those events at its own pace and updates counts. If the consumer is slow, down, or you add ten more consumers later, the redirect path never notices. That decoupling is the entire point of Kafka, and a URL shortener makes it tangible.

  • Django teaches you the request/response core: models, migrations, an API, routing.
  • Postgres is your source of truth — the link table lives here.
  • Kafka teaches producers, topics, partitions, and consumer groups via click events.
  • Kubernetes teaches Deployments, Services, Secrets, ConfigMaps, Jobs, and Ingress by running all four pieces together.

3 · The data model & short codes

A short URL needs: the generated short_code, the original_url, a created_at timestamp, and a click_count. The interesting bit is generating a collision-free short code.

Your task: write a ShortURL model. Auto-generate a 7-character base62 code on first save, retrying if it already exists. Add a DB index on the code (you'll look up by it on every redirect).
Solution — shortener/models.py code
shortener/models.py 
import secrets
import string
from django.db import models

ALPHABET = string.ascii_letters + string.digits  # 62 chars → base62


def generate_short_code(length: int = 7) -> str:
    return "".join(secrets.choice(ALPHABET) for _ in range(length))


class ShortURL(models.Model):
    short_code = models.CharField(max_length=10, unique=True, db_index=True)
    original_url = models.URLField(max_length=2048)
    created_at = models.DateTimeField(auto_now_add=True)
    click_count = models.PositiveIntegerField(default=0)

    def save(self, *args, **kwargs):
        if not self.short_code:
            code = generate_short_code()
            # Retry on the (very rare) collision before inserting.
            while ShortURL.objects.filter(short_code=code).exists():
                code = generate_short_code()
            self.short_code = code
        super().save(*args, **kwargs)

    def __str__(self) -> str:
        return f"{self.short_code} → {self.original_url}"
Why base62 + secrets 627 ≈ 3.5 trillion combinations, so random collisions are astronomically unlikely even with millions of links — yet the code stays short and URL-safe (letters + digits only). secrets (not random) gives cryptographically strong, unguessable codes so people can't trivially enumerate everyone's links.
Gotcha The unique=True constraint is your real safety net. The Python-side retry loop avoids most collisions, but the database constraint is what guarantees uniqueness under concurrency. At massive scale you'd switch to counter-based or Snowflake-style IDs to skip the check entirely.

4 · Create & redirect endpoints

Two routes: POST /api/shorten returns a new short code (JSON, via DRF), and GET /<code> issues a 302 redirect to the original URL.

Your task: build a serializer, a create view, and a redirect view, then wire the URLs. Leave the Kafka call as a TODO for now — you'll fill it in two steps later.
Solution — serializer code
shortener/serializers.py 
from rest_framework import serializers
from .models import ShortURL


class ShortURLSerializer(serializers.ModelSerializer):
    class Meta:
        model = ShortURL
        fields = ["short_code", "original_url", "created_at", "click_count"]
        # The client only supplies original_url; the rest are server-owned.
        read_only_fields = ["short_code", "created_at", "click_count"]
Solution — views code
shortener/views.py 
from django.shortcuts import get_object_or_404, redirect
from django.views import View
from rest_framework import generics

from .models import ShortURL
from .serializers import ShortURLSerializer
# from .kafka_producer import publish_click_event   # <-- enabled in step 6


class CreateShortURL(generics.CreateAPIView):
    """POST /api/shorten  → {original_url} ⇒ {short_code, ...}"""
    queryset = ShortURL.objects.all()
    serializer_class = ShortURLSerializer


class RedirectToOriginal(View):
    """GET /<code> → 302 to the original URL."""
    def get(self, request, short_code):
        obj = get_object_or_404(ShortURL, short_code=short_code)
        # TODO (step 6): publish a click event to Kafka here.
        return redirect(obj.original_url)
Solution — URL routing code
config/urls.py 
from django.contrib import admin
from django.urls import path
from shortener.views import CreateShortURL, RedirectToOriginal

urlpatterns = [
    path("admin/", admin.site.urls),
    path("api/shorten", CreateShortURL.as_view(), name="shorten"),
    # Keep this LAST — it's a catch-all for any single path segment.
    path("<str:short_code>", RedirectToOriginal.as_view(), name="redirect"),
]
Gotcha Order matters. The <str:short_code> pattern greedily matches one segment, so it must come after admin/ and api/shorten or it would swallow them.

6 · Publish click events to Kafka

Now the Kafka half. On every redirect, publish a small JSON event to a topic called url-clicks. This must be fire-and-forget — it cannot block the redirect.

Your task: write a publish_click_event() helper using confluent-kafka, then call it from the redirect view. The producer's bootstrap server must come from an env var.
Solution — producer helper code
shortener/kafka_producer.py 
import json
import os
from confluent_kafka import Producer

# One Producer per process — it manages an internal background thread + buffer.
_producer = Producer({
    "bootstrap.servers": os.environ.get("KAFKA_BOOTSTRAP_SERVERS", "localhost:9092"),
    "client.id": "url-shortener-web",
})

TOPIC = "url-clicks"


def publish_click_event(short_code: str, ip: str | None, user_agent: str) -> None:
    event = {"short_code": short_code, "ip": ip, "user_agent": user_agent}
    try:
        _producer.produce(TOPIC, key=short_code, value=json.dumps(event))
        # poll(0) services delivery callbacks without blocking the request.
        _producer.poll(0)
    except BufferError:
        # Local buffer full — drop the analytics event rather than fail the redirect.
        pass
Why key = short_code Kafka routes all messages with the same key to the same partition, which preserves per-link ordering and lets you scale consumers horizontally while keeping one link's events together. And note we never raise on failure — analytics is best-effort; the redirect is sacred.
Solution — wire it into the redirect view code
shortener/views.py — update RedirectToOriginal 
from .kafka_producer import publish_click_event

class RedirectToOriginal(View):
    def get(self, request, short_code):
        obj = get_object_or_404(ShortURL, short_code=short_code)
        publish_click_event(
            short_code=short_code,
            ip=request.META.get("REMOTE_ADDR"),
            user_agent=request.META.get("HTTP_USER_AGENT", ""),
        )
        return redirect(obj.original_url)
Gotcha We deliberately do not increment click_count here. The whole point is that the web tier only emits the event; the consumer owns the write. Doing both would defeat the decoupling you're trying to learn.

7 · The analytics consumer

A separate long-running process subscribes to url-clicks, and for each event bumps that link's click_count. Build it as a Django management command so it has full ORM access.

Your task: create manage.py consume_clicks that polls Kafka in a loop and applies an atomic F('click_count') + 1 update per message.
Solution — management command code
shortener/management/commands/consume_clicks.py 
# Create the empty package files first:
#   shortener/management/__init__.py
#   shortener/management/commands/__init__.py
import json
import os
from confluent_kafka import Consumer
from django.core.management.base import BaseCommand
from django.db.models import F
from shortener.models import ShortURL


class Command(BaseCommand):
    help = "Consume url-clicks events and update click counts."

    def handle(self, *args, **options):
        consumer = Consumer({
            "bootstrap.servers": os.environ.get("KAFKA_BOOTSTRAP_SERVERS", "localhost:9092"),
            "group.id": "click-processor",          # the consumer GROUP
            "auto.offset.reset": "earliest",        # start from the beginning the first time
            "enable.auto.commit": True,
        })
        consumer.subscribe(["url-clicks"])
        self.stdout.write(self.style.SUCCESS("Listening on url-clicks…"))

        try:
            while True:
                msg = consumer.poll(1.0)
                if msg is None:
                    continue
                if msg.error():
                    self.stderr.write(f"Kafka error: {msg.error()}")
                    continue

                event = json.loads(msg.value())
                # Atomic, race-free increment at the DB level.
                ShortURL.objects.filter(short_code=event["short_code"]).update(
                    click_count=F("click_count") + 1
                )
                self.stdout.write(f"+1 click → {event['short_code']}")
        finally:
            consumer.close()
Why a consumer group group.id="click-processor" is the magic word. Run one replica and it gets every message. Run three and Kafka automatically splits the topic's partitions across them — instant horizontal scaling, no code change. Add a second group later (say, "fraud-detector") and it gets its own independent copy of the whole stream. That's the superpower you're here to feel.
Gotcha F("click_count") + 1 updates in the database, not in Python. If you did obj.click_count += 1; obj.save() you'd lose increments under concurrent clicks (read-modify-write race). The F() expression compiles to SET click_count = click_count + 1, which is atomic.

10 · Deploy Postgres

Postgres is stateful — it needs storage that survives a pod restart. Use a PersistentVolumeClaim, a Deployment, and a Service so other pods can reach it by the name postgres.

Your task: write manifests for a Secret (credentials), a PVC, a Deployment mounting that PVC, and a Service named postgres.
Solution — k8s/postgres.yaml manifest
k8s/postgres.yaml 
apiVersion: v1
kind: Secret
metadata: { name: postgres-secret }
type: Opaque
stringData:
  POSTGRES_DB: shortener
  POSTGRES_USER: postgres
  POSTGRES_PASSWORD: supersecret
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata: { name: postgres-pvc }
spec:
  accessModes: ["ReadWriteOnce"]
  resources: { requests: { storage: 1Gi } }
---
apiVersion: apps/v1
kind: Deployment
metadata: { name: postgres }
spec:
  replicas: 1
  selector: { matchLabels: { app: postgres } }
  template:
    metadata: { labels: { app: postgres } }
    spec:
      containers:
        - name: postgres
          image: postgres:16
          envFrom:
            - secretRef: { name: postgres-secret }
          ports: [{ containerPort: 5432 }]
          volumeMounts:
            - { name: data, mountPath: /var/lib/postgresql/data }
      volumes:
        - name: data
          persistentVolumeClaim: { claimName: postgres-pvc }
---
apiVersion: v1
kind: Service
metadata: { name: postgres }
spec:
  selector: { app: postgres }
  ports: [{ port: 5432, targetPort: 5432 }]
Why a Service = DNS The Service named postgres gives you a stable in-cluster hostname. Any pod can connect to postgres:5432 and Kubernetes load-balances to the right pod — even after the pod restarts with a new IP. That hostname is exactly what you'll put in POSTGRES_HOST next.

11 · Deploy Kafka

Run a single-broker Kafka in KRaft mode (no Zookeeper). For a learning project one broker is plenty; in production you'd use the Strimzi operator instead.

Your task: deploy a one-node Kafka with a Service named kafka, advertising itself as kafka:9092 so in-cluster clients can connect.
Solution — k8s/kafka.yaml manifest
k8s/kafka.yaml 
apiVersion: apps/v1
kind: Deployment
metadata: { name: kafka }
spec:
  replicas: 1
  selector: { matchLabels: { app: kafka } }
  template:
    metadata: { labels: { app: kafka } }
    spec:
      containers:
        - name: kafka
          image: apache/kafka:3.8.0
          ports: [{ containerPort: 9092 }]
          env:
            # Official Apache image: raw property names, KAFKA_ prefix, no _CFG_,
            # and no ALLOW_PLAINTEXT_LISTENER (that's Bitnami-only).
            - { name: KAFKA_NODE_ID, value: "0" }
            - { name: KAFKA_PROCESS_ROLES, value: "broker,controller" }
            - { name: KAFKA_CONTROLLER_QUORUM_VOTERS, value: "0@localhost:9093" }
            - { name: KAFKA_LISTENERS, value: "PLAINTEXT://:9092,CONTROLLER://:9093" }
            # Advertise the Service name so other pods resolve it correctly:
            - { name: KAFKA_ADVERTISED_LISTENERS, value: "PLAINTEXT://kafka:9092" }
            - { name: KAFKA_CONTROLLER_LISTENER_NAMES, value: "CONTROLLER" }
            - { name: KAFKA_LISTENER_SECURITY_PROTOCOL_MAP, value: "CONTROLLER:PLAINTEXT,PLAINTEXT:PLAINTEXT" }
            - { name: KAFKA_INTER_BROKER_LISTENER_NAME, value: "PLAINTEXT" }
            # Single-node replication factors (Bitnami set these implicitly):
            - { name: KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR, value: "1" }
            - { name: KAFKA_TRANSACTION_STATE_LOG_REPLICATION_FACTOR, value: "1" }
            - { name: KAFKA_TRANSACTION_STATE_LOG_MIN_ISR, value: "1" }
            - { name: KAFKA_GROUP_INITIAL_REBALANCE_DELAY_MS, value: "0" }
            # Auto-create url-clicks on first use (fine for dev):
            - { name: KAFKA_AUTO_CREATE_TOPICS_ENABLE, value: "true" }
---
apiVersion: v1
kind: Service
metadata: { name: kafka }
spec:
  selector: { app: kafka }
  ports: [{ port: 9092, targetPort: 9092 }]
Gotcha — advertised listeners The single most common Kafka-on-k8s mistake. A broker tells clients which address to use via its advertised listener. If it advertises localhost, your other pods will try to reach Kafka on their own localhost and fail. It must advertise the Service name: kafka:9092.

12 · Config & Secrets for the app

Give the Django pods their configuration: non-secret values in a ConfigMap, the DB password reused from the Postgres Secret.

Your task: write a ConfigMap with the DB host, Kafka bootstrap, and Django settings the app reads from the environment.
Solution — k8s/app-config.yaml manifest
k8s/app-config.yaml 
apiVersion: v1
kind: ConfigMap
metadata: { name: app-config }
data:
  POSTGRES_HOST: "postgres"               # the Service name from step 10
  POSTGRES_PORT: "5432"
  POSTGRES_DB: "shortener"
  POSTGRES_USER: "postgres"

13 · Run migrations as a Job

Migrations should run once per deploy, not on every pod start. A Kubernetes Job runs a pod to completion — perfect for this.

Your task: write a Job that runs python manage.py migrate using your image, pulling env from the ConfigMap and Secret.
Solution — k8s/migrate-job.yaml manifest
k8s/migrate-job.yaml 
apiVersion: batch/v1
kind: Job
metadata: { name: migrate }
spec:
  backoffLimit: 3
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: url-shortener:dev
          imagePullPolicy: IfNotPresent
          command: ["python", "manage.py", "migrate", "--noinput"]
          envFrom:
            - configMapRef: { name: app-config }
            - secretRef: { name: postgres-secret }
Gotcha — ordering The Job will crash-loop until Postgres is actually accepting connections. That's fine — its backoffLimit retries. For a cleaner setup you'd add an init container that waits for postgres:5432 before migrating. Re-run a Job by deleting and re-applying it.

14 · Deploy the web app

The Django/gunicorn Deployment, fronted by a Service. This is the piece that serves the API and the redirects.

Your task: write a Deployment (2 replicas) + a Service named web, both reading config from the ConfigMap and Secret. Add a readiness probe.
Solution — k8s/web.yaml manifest
k8s/web.yaml 
apiVersion: apps/v1
kind: Deployment
metadata: { name: web }
spec:
  replicas: 2
  selector: { matchLabels: { app: web } }
  template:
    metadata: { labels: { app: web } }
    spec:
      containers:
        - name: web
          image: url-shortener:dev
          imagePullPolicy: IfNotPresent
          ports: [{ containerPort: 8000 }]
          envFrom:
            - configMapRef: { name: app-config }
            - secretRef: { name: postgres-secret }
          readinessProbe:
            httpGet: { path: /api/shorten, port: 8000 }
            initialDelaySeconds: 5
            periodSeconds: 10
---
apiVersion: v1
kind: Service
metadata: { name: web }
spec:
  selector: { app: web }
  ports: [{ port: 80, targetPort: 8000 }]
Why 2 replicas + a probe Two replicas means Kubernetes load-balances redirects across pods and survives one dying. The readiness probe tells Kubernetes "don't send traffic until this pod can actually serve" — so a pod still booting or waiting on the DB won't get requests routed to it.

The payoff With one partition (the dev default) only one of the three pods does work — the others sit idle, ready to take over instantly if it dies (free failover). Bump url-clicks to 3 partitions and all three pods process in parallel. That is the lesson the whole project was built to deliver.

16 · Expose it & test end-to-end

Apply everything in order, then reach the web Service from your laptop with a quick port-forward (simpler than Ingress for a local cluster).

Your task: apply all manifests, port-forward the web Service, and run the same create + redirect test you did locally — but now it's flowing through Kafka and a separate consumer pod inside Kubernetes.
Solution — apply, forward, verify commands

Debugging toolkit When something's wrong: kubectl get pods (status), kubectl describe pod <name> (events — image pull/scheduling issues), kubectl logs <name> (app errors), and kubectl exec -it <name> -- sh (poke around inside). 90% of k8s debugging is those four.

17 · Recap, extensions & cleanup

You just built and orchestrated a four-service distributed system. Make sure you can explain each of these in your own words — that's the real deliverable:

  • Django/DRF: models, migrations, serializers, class-based & API views, URL routing.
  • Postgres: stateful workload, PVC-backed storage, atomic F() updates.
  • Kafka: producers, a topic, keys → partitions, and consumer groups for scaling & fan-out.
  • Kubernetes: Deployment, Service (DNS + load-balancing), ConfigMap, Secret, Job, probes, and one image running two roles.

Extensions to deepen each skill

  • Add a second consumer group (geo-enricher) that looks up IP → country — feel fan-out.
  • Bump url-clicks to 3 partitions and prove parallel consumption with 3 replicas.
  • Add a GET /api/stats/<code> endpoint and a tiny dashboard.
  • Replace the raw Kafka manifest with the Strimzi operator (the production way).
  • Add a HorizontalPodAutoscaler on the web Deployment driven by CPU.
  • Swap port-forward for a real Ingress (install ingress-nginx in kind).
